# Deviation Register

Every entry below is a place where the shipped system's scope or approach
differs from a literal reading of the SRS, together with the SRS text that
authorizes or explains the deviation and the current status. This register
existing and being current is itself the acceptance evidence SRS §6 and
§3.6.7 require — it is not a to-do list.

| # | Requirement(s) | Deviation | Status |
|---|---|---|---|
| 1 | eIDAS QES, TBD-A | v1 signs Advanced Electronic Signatures (AES) only; Qualified Electronic Signatures/Seals (QES) are out of scope. The SRS's own product scope explicitly descopes QES for v1 and defers remote-QES/TSP integration to the party that takes the product to production. | Open by design — see [tbd-resolutions.md](tbd-resolutions.md) TBD-A. |
| 2 | FR-SM-02 (JAdES/CAdES) | One production-grade signature path — PAdES for PDF contracts — is implemented and externally validated (pyHanko, DSS demo webapp). JAdES (JSON) and CAdES (XML/CMS) are not implemented; DCS's contract artifact is always a PDF, so PAdES is the only path exercised in practice. | Roadmap. |
| 3 | §1.2, FR-SM-16 | The signing model is: the organization's AES key lives in an HSM (PKCS#11/SoftHSM2 in dev, a real HSM in production by configuration swap), and every signing act is authorized by a PID-holder completing an OID4VP presentation ceremony via EUDIPLO — "the company will sign … via OID4VP, with the user signing it using their AES PID," per the SRS. A separate user-held-key AES flow via an OCM/PCM signer API (the SRS's "such as" wording for FR-SM-16) is not implemented; the HSM-plus-PID-ceremony model is the one production-grade path. | Roadmap (alternative signer model). |
| 4 | §1.2 vs §3.1.1 (contract state model) | The SRS's own state lists are internally inconsistent between §1.2 (narrative lifecycle) and §3.1.1 (IR-CWE-05..10 state names). The implemented state machine (`DRAFT → OFFERED → NEGOTIATION → ACCEPTED/APPROVED → SIGNED → ACTIVE → …`, plus `REJECTED`/`WITHDRAWN`/`REVOKED`) is the reconciliation of both, with `APPROVED` declared equivalent to §1.2's "accepted." See ADR-2. | Resolved — reconciliation is the shipped state machine. |
| 5 | DCS-OR-C2PA-008 | C2PA manifests are embedded in the PDF using a DCS-specific transport (JUMBF boxes attached via the PDF's embedded-file mechanism, not the C2PA-native asset-embedding profile for PDF, which does not yet exist as a ratified spec). The **remote manifest** path — a `dcterms:provenance` link in the PDF's XMP metadata pointing to an externally hosted manifest — is the standards-compliant fallback and is fully implemented; general-purpose C2PA tooling (c2patool, verifier UIs) that cannot parse the embedded transport can still verify provenance via the remote link. | Resolved — remote manifest path shipped (D3). |
| 6 | ODRL profile (FR-PACM-*) | Contract policies are emitted as a well-formed `odrl:Set` under a DCS-published profile, with concrete rule types (`OdrlDuty`, `OdrlPermission`, `OdrlProhibition`) and server-side constraint satisfaction enforcement (a security-relevant control, not optional — see F2). The `odrl:Offer` → `odrl:Agreement` two-party upgrade flow and compound (AND/OR-nested) constraint expressions are not implemented; every shipped policy is a single-party `odrl:Set` with a flat constraint list. | Roadmap (Offer/Agreement upgrade, compound constraints). |
| 7 | IR-HI-02, IR-HI-03 | FIDO2/WebAuthn step-up authentication and TPM-backed key sealing are not implemented. IR-HI-01 (standardized key-custody interface) is satisfied via PKCS#11/SoftHSM2 in dev; swapping in a real HSM or a TPM-backed provider is a configuration change against the same PKCS#11 interface, not a code change. | Roadmap. |
| 7a | FR-SM-03, FR-SM-04, TBD-B | Identity (PID) binding under every signature is shipped. Power-of-Attorney (PoA) credential *presentation* — requesting and receiving a PoA credential via DCQL/OID4VP — is implemented (`backend/internal/auth/oid4vp/poa.go`). The PoA *chain-walk* (verifying the PoA issuer's own authority back to a trust anchor, rather than just checking the PoA credential's own signature) is not implemented — see [tbd-resolutions.md](tbd-resolutions.md) TBD-B, which is the SRS's own acknowledgment that the wallet-side PoA ecosystem (XFSC PCM) is not yet confirmed available to test against. | Partial — PoA presentation shipped; chain-walk to trust anchors is roadmap. |
| 7b | NFR-BR-01 | "Two-factor authentication" is satisfied by wallet-based verifiable-credential presentation: possession of the wallet plus the wallet's own unlock factor (PIN/biometric) constitute the two factors. There is no separate DCS-side TOTP/WebAuthn step. | Resolved — satisfied via the wallet's own two-factor unlock, not a DCS-side second factor. |
| 8 | Login hardening (`backend/internal/auth/oid4vp/verify.go`) | Of the three login-path hardening items originally flagged as stubs: **credential revocation checking is implemented and wired into every verification call** (`checkStatusList`, called from both `Verify` and `VerifyPID`). Two remain open: (a) issuer→role authority — `evaluateLoginRoles` validates that a disclosed role string is a recognized `UserRole`, but does not check that the credential's issuer is authorized to assert that specific role; (b) DCQL query results are not cross-checked claim-by-claim against the query that was sent (the query is built and transmitted, but the response is trusted rather than re-validated against it). | Partial — revocation check resolved; issuer-authority and DCQL-result enforcement remain open. |
| 9 | §3.6.8 vs §2.8 | §3.6.8 refers to "QES signing" for the acceptance demo while §2.8 explicitly descopes QES for v1. This is a direct SRS-internal contradiction, flagged to ECO rather than silently resolved one way. The PoC demonstrates AES per §2.8's product scope. | Open, flagged to ECO — not a DCS defect. |
| 10 | PC-01, Appendix D | XFSC component posture: the Federated Catalogue, XFSC Status List Service, and ORCE (orchestration engine, Node-RED-webhook-compatible per IR-SI-02) are retained and integrated as specified. The Appendix D "crypto-provider" (described there as "essential crypto parts") is replaced by PKCS#11/HSM key custody — defended under IR-HI-01's "standardized interfaces" wording; the crypto-provider chart was fully removed (not left as a parallel unused path — clean-product sweep item 3). | Resolved (documented substitution) — crypto-provider swapped for PKCS#11, not dropped. |
| 11 | IR-CWE-10 | Approved contracts are not automatically registered in the Federated Catalogue. Template→catalogue registration exists (`backend/internal/templatecatalogueintegration/`) and is genuinely reusable infrastructure, but its self-description builder (`fctemplatebuilder.go`) is hard-bound to the template JSON-LD shape (fixed `@type`, template-specific context, template ID-mapping) — reusing it for contracts would fake reuse rather than deliver it. A contract-specific self-description builder is new work. The natural hook point, if built, is the `APPROVED` transition in `contractworkflowengine/command/approve.go`, which already performs archive/IPFS/notary/TSA side effects at that point. | Deferred — infrastructure exists, contract-specific builder is roadmap. |
| 12 | Long-term signature preservation | v1 ships PAdES-B-T (signing-time RFC 3161 timestamps) plus versioned key rotation with old-key validation retained (A5's revocation/rotation drill). PAdES-B-LTA (periodic archive re-timestamping — re-sealing archived signatures under fresh algorithms/timestamps every 1–2 years so they remain verifiable as older algorithms weaken) is designed and documented here and in the admin guide's key-rotation section, but its automated execution is an operations-team responsibility, not shipped code. | Roadmap (automation); design documented. |
| 13 | IR-SI-10, IR-CI-08 | Contracts are signed in-house: the backend's PKCS#11-held key produces the CMS/COSE bytes and pdf-core assembles the PAdES envelope (B1/B2) — no remote Digital Signature Service (DSS) or Trust Service Provider executes the signing operation. The **EU DSS** library (the SRS's own `[DSS]` reference) is used as an independent third-party *validation* reference for the PAdES output, confirming the produced signatures conform without depending on DSS for signing itself. The formerly in-repo `dss` package name was a misnomer (it was the PDF-core signing client, not the EU Digital Signature Service) and has been renamed away; `signingmanagement/dss` no longer exists in the codebase. Remote-signing-executor integration (a real DSS or TSP performing the signing operation on DCS's behalf) is roadmap. | Resolved (misnomer) / Roadmap (remote signing executor). |

## Owner action required

Per the SRS, this register is only effective as acceptance evidence if the
scope items above — particularly #1 (QES), #9 (the §3.6.8/§2.8
contradiction), and #11 (catalogue registration) — are raised with the
ECO/acceptor **before** formal acceptance, not discovered afterward.
